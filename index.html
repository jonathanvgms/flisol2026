Qué es (y qué no) el vibe coding

El término lo acuñó Andrej Karpathy a principios de 2025: entregarse a las vibras y aceptar lo que el agente produce sin revisarlo a fondo.

Su valor real
  • Velocidad y exploración: idea → prototipo en minutos
  • Baja la barrera de entrada (no hace falta ser programador)
  • Ideal para descartables, demos y probar hipótesis
Su costo oculto
  • No queda registro de las decisiones que se tomaron
  • Aceptás código sin entenderlo: errores y vulnerabilidades que no ves
  • No hay un contrato contra el cual verificar lo generado

No es un anti-patrón: es una herramienta. El problema empieza cuando la usás fuera de su rango.

Entonces.. La velocidad aumentó; el problema ahora es el control.

Ejemplo · applyCoupon

El prompt de un martes a las 5pm

Estoy armando el checkout de un e-commerce. Necesito una función JS: applyCoupon(cartTotal, coupon) que devuelva el total con el descuento aplicado. El cupón puede ser por porcentaje o por monto fijo. Hacelo simple, después lo refino.
Ejemplo · applyCoupon

Lo que el agente devuelve

function applyCoupon(cartTotal, coupon) { if (coupon.type === 'percentage') { return cartTotal - (cartTotal * coupon.value / 100); } return cartTotal - coupon.value; }
applyCoupon(100, { type:'percentage', value:20 }) // → 80 ✅

Tres maneras de romperlo

1

Total negativo

applyCoupon(30, { type:'fixed', value:50 }) // → -20

2

// La función no chequea fecha. Un cupón de Black Friday // sigue funcionando en marzo.
Nunca dije que los cupones venzan.
3

Reaplicación infinita

let t = 100; for (let i = 0; i < 5; i++) t = applyCoupon(t, { type:'percentage', value:50 }); // → 3.125
Nunca dije si se acumulan.

Lo mismo, pero spec-driven

# Feature: Aplicar cupón al carrito ## Reglas - Tipos: PERCENTAGE, FIXED_AMOUNT - Validar fecha de vencimiento (UTC) - Validar monto mínimo del carrito - Un solo cupón activo por carrito - El descuento no puede dejar el total bajo 0 - Redondeo HALF_UP a 2 decimales - Validar límite de uso por usuario ## Criterios de aceptación - Carrito $100 + cupón 20% → total $80.00 - Carrito $30 + cupón fijo $50 → total $0.00 (no negativo) - Cupón vencido → error VOUCHER_EXPIRED - Usuario ya usó el cupón → error VOUCHER_ALREADY_USED

El muro

El vibe coding deja de ser confiable pasada cierta complejidad: el agente empieza a adivinar requisitos que nunca enunciaste.

~500 líneas es el umbral que se suele citar.
No es un límite exacto: es donde lo implícito se acumula y empieza a romper.
1

Requisitos implícitos

El agente inventa las decisiones que no tomaste: bordes, validaciones, reglas de negocio. Como en applyCoupon.

2

Deriva de contexto

Entre sesiones el contexto se contamina y se pierde. Lo que "sabía" ayer hoy ya no está, y las decisiones cambian sin avisar.

3

Sin contrato

No hay una fuente de verdad contra la cual verificar. "Parece andar" es lo único que podés comprobar.

La mayor parte del tiempo no estaba arreglando bugs: estaba arreglando decisiones poco claras que nunca tomé de forma explícita. — parafraseando a un dev sobre vibe coding

Modo plan y build

Plan = pensar sin tocar código

Pensado para analizar, proponer cambios y armar un plan sin modificar el código fuente.

Build = ejecutar cambios reales

Modo de implementación. Tiene las herramientas habilitadas para editar archivos, ejecutar comandos y hacer trabajo de desarrollo real sobre el proyecto.

Spec Driven Development

Definís comportamiento, restricciones y criterios de aceptación antes; el agente implementa contra ese contrato. Martin Fowler y ThoughtWorks ya lo documentan: pasó de metodología emergente a casi estándar de industria.

La spec como fuente de verdad

  • Especificación clara de lo que el sistema debe hacer
  • Describir intención, comportamiento y alcance, y sirve para alinear a todos antes de construir

Implementar y verificar contra la spec

  • Guía la implementación y también para verificar si lo construido responde a lo pedido
  • El equipo escribe especificaciones claras del comportamiento esperado y luego genera, implementa o verifica código contra esas especificaciones

La spec como documento vivo

  • Una buena spec no se escribe una vez y se momifica
  • Se puede modificar, extender o reducir su alcance. Si la spec es muy grande posiblemente deba ser reducida en varias specs
  • Matiz: las specs estáticas se vuelven caras y derivan; las vivas se sincronizan con el código

De autocompletado a ingeniero

El truco conceptual de SDD: dejás de pedirle qué tipear y empezás a pedirle qué comportamiento garantizar.

⚙️ Instrucción técnica (vaga)
Usá PostgreSQL para guardar la actividad de los usuarios.

Describe la implementación, no el resultado. El agente completa los huecos adivinando. No hay nada que verificar.

🎯 Criterio conductual (verificable)
Como admin puedo consultar la actividad de usuarios de los últimos 90 días y recibir resultados en menos de 2 s.

Define comportamiento observable y restricciones. Es un contrato: o se cumple, o no. La base de datos es un detalle.

El workflow de 3 fases

Las herramientas le ponen nombres distintos, pero la estructura es siempre la misma. Un modelo mental portable, independiente de la herramienta.

FASE 1

Definir el qué

RequirementsAWS Kiro SpecifyGitHub Spec Kit

Comportamiento, restricciones y criterios de aceptación. Existe notación formal para esto (EARS).

FASE 2

Diseñar el cómo

DesignAWS Kiro PlanGitHub Spec Kit

Arquitectura, modelo de datos y decisiones técnicas, derivadas de la spec.

FASE 3

Partir en tareas

TasksAWS Kiro TasksGitHub Spec Kit

Cambios chicos y revisables que el agente implementa y verifica contra el contrato.

Implementación SDD

  • Escribir la spec: alcance, comportamiento esperado, restricciones y criterios de aceptación
  • Documentar el diseño: arquitectura, modelo de datos, APIs, impacto y riesgos
  • Partir el trabajo en tareas chicas y las implementás en cambios pequeños y revisables
  • Registrar decisiones importantes en ADRs (Architecture Decision Record)
  • Revisar contra la spec, no solo contra el código
  • Actualizar la spec después de su implementación

SDD busca trazabilidad y precisión al construir con IA

OpenSpec

Alinear antes de programar

  • Equipo humano y agente acuerdan qué se va a construir antes de tocar código
  • "Agree before you build", para que los requisitos se tengan en cuenta

Convertir la intención en artefactos explícitos

  • En vez de depender de prompts sueltos, el cambio se estructura con artefactos concretos: proposal, specs, design y tasks
  • De "pedido informal" a "trabajo trazable y ejecutable"

Cerrar el ciclo con trazabilidad y fuente de verdad

  • El desarrollo no termina al implementar: el cambio aprobado se archiva, se integra a las specs principales y queda historial auditable (archivos markdown)

No alcanza con pedir bien: hay que construir con trazabilidad

El trade-off es honesto: más trabajo al principio a cambio de menos sorpresas después. No vale la pena para un script rápido; es esencial para algo que va a producción. La clave es saber cuándo usar cada modo.
