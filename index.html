Vibe Coding

"Es como pedirle a alguien que te construya una casa describiéndola de memoria y después descubrir que no hay cimientos"

Lo bueno
  • Velocidad extrema para prototipos
  • Accesible para no-programadores
Lo no tan bueno
  • No queda registro de las decisiones tomadas
  • Se puede usar código sin entender su funcionalidad, generando errores o vulnerabilidades no detectadas

Modo plan y build

Plan = pensar sin tocar código

Pensado para analizar, proponer cambios y armar un plan sin modificar el código fuente.

Build = ejecutar cambios reales

Modo de implementación. Tiene las herramientas habilitadas para editar archivos, ejecutar comandos y hacer trabajo de desarrollo real sobre el proyecto.

Spec Driven Development

La spec como fuente de verdad

  • Especificación clara de lo que el sistema debe hacer
  • Describir intención, comportamiento y alcance, y sirve para alinear a todos antes de construir

Implementar y verificar contra la spec

  • Guía la implementación y también para verificar si lo construido responde a lo pedido
  • El equipo escribe especificaciones claras del comportamiento esperado y luego genera, implementa o verifica código contra esas especificaciones

La spec como documento vivo

  • Una buena spec no se escribe una vez y se momifica
  • Se puede modificar, extender o reducir su alcance. Si la spec es muy grande posiblemente deba ser reducida en varias specs

Implementación SDD

  • Escribir la spec: alcance, comportamiento esperado, restricciones y criterios de aceptación
  • Documentar el diseño: arquitectura, modelo de datos, APIs, impacto y riesgos
  • Partir el trabajo en tareas chicas y las implementás en cambios pequeños y revisables
  • Registrar decisiones importantes en ADRs (Architecture Decision Record)
  • Revisar contra la spec, no solo contra el código
  • Actualizar la spec después de su implementación

SDD busca trazabilidad y precisión al construir con IA

OpenSpec

Framework para estructurar cambios antes de implementarlos

  • Framework para organizar cambios con proposal, specs, design y tasks
  • Convierte intención en artefactos trazables
  • Ayuda a trabajar con IA sin depender de prompts sueltos

OpenSpec

Alinear antes de programar

  • Equipo humano y agente acuerdan qué se va a construir antes de tocar código
  • "Agree before you build", para que los requisitos se tengan en cuenta

Convertir la intención en artefactos explícitos

  • En vez de depender de prompts sueltos, el cambio se estructura con artefactos concretos: proposal, specs, design y tasks
  • De "pedido informal" a "trabajo trazable y ejecutable"

Cerrar el ciclo con trazabilidad y fuente de verdad

  • El desarrollo no termina al implementar: el cambio aprobado se archiva, se integra a las specs principales y queda historial auditable (archivos markdown)

No alcanza con pedir bien: hay que construir con trazabilidad
